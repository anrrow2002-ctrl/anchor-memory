# Anchor Memory 0.9.11 升级说明

## 为什么升级

0.9.10 实际只有一个隐藏分支：点击“全量合并/重写”且没有新增材料时，才尝试重写最近一次累计合并。它没有独立入口，也没有分段锚点重写，因此用户无法明确控制，更不能重写指定历史记录。

## 0.9.11 新增的明确操作

总览页现在有四个独立按钮：

- **生成新锚点**：把当前尚未锚定的有效摘要生成一条新分段锚点。
- **立即全量合并**：把当前尚未并入累计历史的材料生成一次新合并。
- **重写最近锚点**：按最近锚点原本覆盖的楼层重新生成，编号不变。
- **重写最近全量合并**：按最近合并原本的累计边界重新生成，编号不变。

“记忆库”中还支持：

- 每张卡片直接点击“重写此分段锚点 / 重写此全量合并”；
- 选中任意历史记录后点击“重写选中锚点/合并”。

## 重写规则

### 分段锚点

插件重新读取该锚点原本覆盖的逐楼摘要；缺失摘要时读取对应保底原文。重写保留原 ID、编号和覆盖楼层，不会新增一次锚点。

若这个锚点已经进入累计合并，插件会从第一个依赖它的合并开始回滚，防止新锚点继续搭配旧累计历史。达到合并阈值后，后台会按顺序重新生成。

### 全量合并

插件使用“上一次累计历史 + 这一次合并周期的原始材料”重新生成，而不是只润色现有正文。重写保留原 ID、编号和覆盖边界。

重写较早的全量合并时，其后的合并会回滚并等待顺序重建，因为后续合并都依赖被改写的历史版本。

## 数据安全

- 模型返回成功且格式有效后才提交新正文。
- 生成失败、结果过短或中途切换聊天时，旧版本保持不变。
- 不清空逐楼摘要、人物关系、人物/物品/场景索引、档案或 API 配置。
- 本次不改变聊天数据结构，直接覆盖安装即可。

## 注意

重写会调用副 API。重写旧记录后若触发依赖合并重建，后续重建也会产生相应 API 调用。
