# 0.8.0 长聊天性能与持久化审计

- SillyTavern 的聊天 metadata 仍随聊天整体保存；插件无法把底层聊天文件改成数据库增量写入，因此本版从“减少 metadata 体积”和“减少保存次数”两侧处理。
- 每楼历史注入记录从完整正文改为哈希、短预览与来源引用；旧记录迁移时删除重复全文。
- 向量主存储迁移到浏览器 IndexedDB，metadata 只留引用；导出、导入和记忆档案不携带大体积向量，可按需重建。
- 人物关系回滚历史使用 10 回合检查点和逐楼差量，仍支持从高楼层删除回低楼层后的状态恢复。
- 聊天卡片渲染限制在可见区域与最近楼层；工作台逐楼摘要分页。
- 发送前裁剪只遍历严格历史窗口计算出的待隐藏消息，不再对所有 AI 楼层逐个搜索请求数组。
- `saveMemory(true)` 不再让每个调用立即直写；900 ms 窗口内的关键保存合并为一次 flush，页面隐藏时再做最终落盘尝试。

# 0.7.9 固定人物关系表安全审计

- 关系表使用聊天 `chatMetadata.anchorMemory.relationshipTable` 持久化，不依赖外部 SQL 服务；表结构固定为“名称 / 过去 / 发展 / 当前”。
- 第一行永久绑定 `{{char}}`，显示与后台提示时按当前 SillyTavern 对话解析；用户可添加、改名或删除其余固定行。
- 副 API 必须返回全部固定行，且不得含额外行、重复行或陌生名称；否则整次关系更新作废，活动表不会被部分覆盖。
- 增量更新不会用“未明”覆盖已经确认的关系内容；全量重建则从空白固定模板重新计算，避免删楼后沿用未来状态。
- 每 10 个 AI 回合保存完整检查点，中间楼层只保存发生变化的关系行。编辑、swipe、regenerate、删除原楼或手动修改摘要时，按检查点与差量恢复目标楼状态，再根据仍有效 Godlog 复核。
- 从第 68 楼删除到第 23 楼时，24 楼及之后的关系快照被移除；重建输入只来自当前仍存在的剧情摘要。重建成功前旧关系不注入主模型。
- 关系更新与重建均在临时副本中解析，完整校验通过后才原子替换；API 失败、截断、缺行或多行均保留固定名单与最后安全数据。
- 当前记忆 JSON 保留关系检查点与差量；紧凑记忆档案只保存当前关系状态，不复制回滚历史。

# 0.7.8 持久化安全审计

- 所有非用户主动“清空当前记忆”的路径均禁止把 `data.codex` 赋为空对象。
- `markCodexDirty()` 仅保存备份、标记待重建并清理处理指纹，不删除人物、物品或场景正文。
- `rebuildCodexFromGodlogs()` 在临时对象中解析与校验副 API 返回；成功后调用 `commitCodexReplacement()` 原子替换。
- 无有效摘要、API 失败、截断或空表返回时保留旧索引。
- 升级时使用 `sourceHashSchema` 原地迁移来源指纹，避免算法变化触发全量误失效。
- 待重建索引不会进入主模型注入链路，但仍可在面板查看、导出或恢复。

# Anchor Memory 0.7.1 逻辑与缺陷审计

## 结论

0.6.x 的主要问题不是单一保存函数，而是消息身份、隐藏状态、派生记忆依赖和边界计算彼此耦合。0.7.1 将数据拆成“源楼层 → 逐楼摘要 → 15 回合锚点 → 100 回合累计锚点”四层，并为每一层保存可追溯来源。

无法诚实承诺在所有 SillyTavern 版本、主题和第三方扩展组合中不存在任何 bug；本次完成的是完整静态审查、核心状态机重构和多组隔离模拟测试。

## 已修复的高风险问题

### 1. 刷新后重复生成

旧逻辑把通用 `MESSAGE_RENDERED` 当成新剧情事件。刷新页面时历史楼层会重新渲染，从而反复排队。

修复：只有角色新回复事件负责检查生成任务；通用渲染事件仅刷新 UI。任务入队前还会检查是否真的存在缺失摘要、完整 15 回合批次或完整 100 回合批次。

### 2. 隐藏楼层导致摘要指纹变化

旧逻辑生成锚点后隐藏源楼层，刷新时又在计算正文指纹时跳过隐藏消息，结果同一条正文得到不同指纹，被误判为编辑或重 roll。

修复：隐藏只是一种 UI/上下文策略，永远不参与源文本指纹。

### 3. metadata 尚未恢复就读取空记忆

插件可能早于聊天 metadata 完成恢复。旧逻辑会在临时默认对象上判断“所有摘要都缺失”。

修复：没有持久化聊天上下文时禁止读取、生成和保存；`CHAT_CHANGED` 后轮询等待 metadata 就绪。

### 4. 防抖保存尚未落盘就刷新

旧逻辑只调用防抖保存。生成结束后立即刷新或切换聊天时，数据可能尚未写入。

修复：摘要、锚点、合并、导入和回滚等关键操作执行立即 flush；页面隐藏时再次保存。

### 5. 隐藏后楼层编号重排

旧逻辑按可见消息重新编号，隐藏历史后 Godlog 编号会重置或重复。

修复：编号始终基于包含隐藏消息的完整 AI 回合序列，系统消息永不计入。

### 6. 重 roll 后旧摘要仍存在

旧逻辑没有完整删除 `messageGodlogs`、`messageRecalls`、向量和派生锚点；部分旧卡片仍显示，旧记忆仍可能注入。

修复：源正文指纹变化或源消息消失时，删除该楼摘要、楼层 UI 索引、注入记录和向量，并进行依赖回滚。

### 7. 只把锚点标记 stale，没有级联撤销累计合并

100 回合合并是累计产物。早期来源变化后，只标记某个 15 回合锚点过期不足以保证后续累计总结正确。

修复：删除包含变化楼层的 15 回合锚点；删除直接依赖它的累计合并，并删除其后的全部累计合并。变化之前且不依赖该楼的旧累计锚点可保留。

### 8. 人物和物品表出现“幽灵事实”

增量表格可以添加事实，却无法可靠地从累计 Markdown 中减掉某次重 roll 前的事实。

修复：发生源楼变动时清空状态表并标记 dirty，再根据仍有效的逐楼摘要一次性重建。人物/物品/场景表默认不注入主模型。

### 9. 只注入最新一个 15 回合锚点

旧逻辑会丢失最近一次 100 回合合并之后、更早但仍未合并的 15 回合锚点。

修复：注入“最新累计锚点 + 它之后的全部有效 15 回合锚点”。

### 10. 只注入少量最近摘要

旧逻辑只取最后几条摘要，导致上一锚点之后尚未成批的中间剧情可能缺失。

修复：注入上一锚点之后全部有效逐楼摘要，但排除最近 3 个原文窗口，避免同一逐楼摘要与对应正文重复。

### 11. 100 回合实际在 105 回合触发

旧逻辑按 15 回合锚点数量推算合并，因此只能在 15 的倍数附近合并。

修复：100 回合合并直接按连续有效 AI 回合键计数，精确覆盖 1—100、101—200 等周期。

### 12. 15 回合锚点跨越 100 回合边界

例如已经有 90 回合时，下一批会生成 91—105 锚点，与 1—100 合并重复。

修复：100 回合边界优先。合并使用 1—90 的完整锚点加 91—100 的逐楼摘要；101—105 留待与 106—115 组成干净的新锚点。升级时也会清理旧跨界锚点。

### 13. 缺失摘要时用原文静默凑锚点

这会让“每 15 个摘要”变成不确定边界，也让重建结果难以复现。

修复：只接受连续 ready 摘要。遇到缺失楼层立即停止边界推进；主模型暂时保留该楼原文，直到摘要成功。

### 14. UI 失败反过来污染核心任务状态

锚点已经保存后，如果刷新面板报错，旧逻辑可能进入生成失败分支并再次创建重复锚点。

修复：核心落盘与 UI 刷新隔离；预览失败只记录警告，不把成功的记忆任务改判为失败。

### 15. stale 最新记录仍可能被选中

旧版 `latestAnchor` / `latestMerge` 可能返回已经过期或停用的记录。

修复：迁移时清除 stale/inactive 派生项，选择器只读取有效数组并按来源时间排序。

### 16. 系统消息被当作剧情楼层

旧版某些 `includeHidden=true` 分支会把系统消息包含进扫描结果。

修复：系统消息在所有剧情楼层扫描中无条件排除。

## 当前确定的数据流

```text
用户输入 + AI回复
        ↓ rawHash/sourceKey
逐楼摘要 Godlog
        ↓ 连续15条 ready
15回合新锚点
        ↓ 连续100个AI回合
累计历史锚点
```

主模型上下文：

```text
最新累计历史锚点
+ 累计锚点之后的全部15回合锚点
+ 尚未成锚且早于最近3回合的逐楼摘要
+ 最近3个AI正文及其用户输入
```

人物、关系、物品和场景表：

```text
当前状态快照（可查看、可重建、默认不注入）
```

## 重 roll / 编辑 / 删除回滚规则

源楼变化时：

1. 删除对应 Godlog；
2. 删除该楼 UI 摘要卡索引和上一轮注入记录；
3. 删除包含该楼的 15 回合锚点；
4. 删除包含该楼或依赖该锚点的累计合并；
5. 因累计合并存在前后依赖，继续删除它之后的累计合并；
6. 解除不再有效的隐藏状态；
7. 清空不可逆增量状态表并按存活摘要重建；
8. 新正文稳定后重新生成摘要，再恢复精确边界。

## 测试结果

本次 0.7.1 追加验证：

```text
node --check index.js
PROMPT PRUNE TEST PASSED
PREFIX + CONTENT ARRAY TEST PASSED
```

覆盖内容：

- 7 个 AI 回合时只保留最后 3 个 AI 回合正文；
- 更早的 4 个 user/assistant 回合从最终 Prompt 数组删除；
- 带人物名前缀的字符串消息可以剥离旧正文；
- `content` 数组形式的多模态消息可以剥离旧正文；
- `{ chat, dryRun }` 与直接数组事件载荷可识别；
- JavaScript 语法检查通过。

0.7.0 已有的锚点边界、100 回合合并和重 roll 级联测试逻辑未删除；但真实环境仍需用用户当前 SillyTavern 版本做最终生成验证。

## 仍需注意的边界

1. **最近 3 回合与最新锚点最多有 3 回合语义重叠。** 这是同时满足“每 15 回合立即锚定”和“保留最近 3 回合正文”的必然结果，不是重复注入逐楼摘要造成的 bug。
2. **非常旧且完全没有 `send_date`、message id 等持久标识的聊天**只能使用楼层位置作为后备身份。删除前方楼层时可能触发较大范围的安全回滚和重生成，但不会继续使用无法确认的旧记忆。
3. **副 API 不可用时**，缺失摘要会阻塞 15/100 边界；插件保留原文作为唯一事实来源，不会伪造或跳过该楼。
4. **真实环境兼容性**仍需在用户实际 SillyTavern 版本、所用主题、其他扩展和 API 返回格式下验证。隔离测试无法覆盖所有 DOM 和事件时序差异。
5. **跨聊天档案**属于手动携带的历史分支。加载前应备份当前聊天，避免把不相干世界线混入同一累计历史。

## 0.7.1 追加审计：最终 Prompt 正文泄漏

发现 0.7.0 的压缩逻辑主要依赖“把旧 chatHistory 正文替换成 Godlog”，同时又在独立 system 记忆块中注入同一批 Godlog。该方案既可能重复，也受不同 Prompt Builder 的包装方式影响。0.7.1 在最终 `CHAT_COMPLETION_PROMPT_READY` 消息数组上删除已被压缩记忆覆盖的旧 user/assistant 项，只保留最近 3 个 AI 回合。缺摘要的旧回合不会删除。 Prompt List 使用 dry run，而 0.7.0 在 `dryRun=true` 时直接返回，这正是截图中旧正文仍完整出现的直接原因；0.7.1 会在 dry run 中执行相同压缩，但不持久化注入记录。

此外，0.7.0 的自动隐藏只直接写 `is_hidden`，而 SillyTavern 官方隐藏接口使用 `is_system`。0.7.1 改用 `hideChatMessageRange`，并保留插件来源标记以支持重 roll、删除和回滚后的自动取消隐藏。


## 0.7.5 追加审计：三轮正文必须是硬上限

0.7.1 的最终 Prompt 修复仍把“有有效逐楼摘要或已被锚点覆盖”作为删除旧正文的前提。其直接后果是：摘要缺失、失败、待刷新或尚未进入 15 回合锚点的旧楼会继续以完整正文留在 `chat history`，因此设置值 3 实际只是“至少保留最近 3 个 AI 正文”，不是“最多只发送 3 个”。

0.7.5 将两条链路彻底解耦：

1. 摘要/锚点链路只负责压缩记忆是否完整；
2. 请求可见性链路只按真实聊天中的 AI 楼位置计算最近窗口。

第 4 个及更早的 AI 楼以及相应旧用户输入，无论摘要状态如何，都进入隐藏集合；摘要缺失只会产生显式记忆缺口提示，不再触发完整正文回退。持久隐藏在用户消息写入和请求构造前同步，最终消息数组又在生成拦截器与 `CHAT_COMPLETION_PROMPT_READY` 阶段复核。

该设计参考了 Horae 的关键原则：按 AI 消息位置解析 `keepStart`，并将边界以前的连续消息作为隐藏候选，而不是用摘要成功与否决定可见性。Anchor Memory 仍保留最近 3 个 AI 回合对应的用户输入，以维持完整的局部问答上下文。


## 0.7.6 追加审计：稳定锁与状态索引拆分

0.7.5 的稳定判断仍有三处全局耦合：流事件可误置全局生成锁；单个 `settleTimer` 会被后来的楼层覆盖；历史楼也被最新主生成状态阻塞。0.7.6 改为按源楼 key 管理独立计时器，并让生成状态只约束最新 AI 楼。手动重跑在主模型实际未生成时可绕过错误生命周期锁，最终提交仍需通过源正文哈希复核。

状态索引也由一个整表开关拆开：人物按最近对话中的姓名/称谓筛选对应行；重要物品、伏笔和内部梗使用独立持续注入开关；动态召回只负责额外旧剧情片段。
