# Anchor Memory 0.9.1：全数据流、召回与人物追踪重构报告

## 1. 本次目标

0.9.1 不再把性能、摘要、锚点、人物索引和召回当成互不相关的功能，而是统一为一条可回滚、可校验、非阻塞的数据流：

```text
用户消息
  ├─ 立即启动动态召回预取（不阻塞主模型）
  └─ 主模型生成
       ├─ 流式阶段：240ms 节流，只探测最后一楼，不扫描全聊天
       └─ AI 消息完整落盘
            ├─ 1.8s 稳定判定 + 正文哈希复核
            ├─ 副 API 生成本楼 Godlog
            ├─ 代码校验 XML、楼层键和正文哈希
            ├─ 副 API 增量更新关系/人物/NPC/物品/场景索引
            ├─ 代码层白名单清洗后事务式提交
            ├─ 每 15 个有效 AI 回合生成一次新增锚点
            ├─ 每 100 个有效 AI 回合生成一次累计历史锚点
            ├─ 向量异步写入 IndexedDB
            └─ 合并保存请求、按需刷新可视 UI

下一次主模型组 Prompt
  ├─ 保留最近 3 个 AI 回合原文及对应用户输入
  ├─ 更早原文从最终请求数组移除
  ├─ 固定拼接一至五节
  ├─ 补充尚未进入 15 回合锚点的逐楼摘要
  ├─ 读取已经预取完成的向量结果；未完成则即时关键词回退
  └─ 一次性插入完整六节系统记忆块
```

## 2. 主模型收到的固定结构

0.9.1 在所有后端上都使用同一个完整系统块，顺序固定，不再把核心记忆和召回拆成两个可能乱序的扩展 Prompt：

```text
锚点记录

【一. 人物关系】
### 人物名
过去：一句话概括初始状态
发展：一句话概括关系推进过程
当前：一句话概括目前的核心关系与拉扯状态

【二. 锚点事件】

**1. 历史锚点简述（100楼的全量）：**
* **[时间段] - [事件名称]：** 起因 → 核心冲突 → 结果/影响。关键对话注明说话人并保留原话。

**2. 本次新增锚点（15楼的锚点）：**
* **[时间] - [事件名称]：** 地点/起因/人物/详细过程/重要物品/结果与影响/核心对话原话。

【三. 追踪主角 动态演变（核心转变）】
人物纪要表；只允许追踪白名单角色。

【四. 匹配到的出场人物库】
| 角色名 | 身份/标签 | 当前状态与核心作用 | 与玩家的关系 | 与追踪主角的关系 |
| :--- | :--- | :--- | :--- | :--- |

【五. 重要道具、梗与核心细节】
| 物品/细节/内部梗 | 绑定人物 | 核心象征意义与影响 |
| :--- | :--- | :--- |

【六. 逐楼摘要与动态召回】
### 逐楼摘要（尚未进入15楼锚点）
...
### 动态召回（与当前输入相关的旧楼细节）
...
```

固定层级之间的边界：

- 100 回合累计历史锚点：固定必注入，只保留最新一份累计结果。
- 15 回合锚点：固定必注入，只注入最新累计锚点之后尚未合并的锚点。
- 逐楼摘要：只补“没有进入 15 回合锚点、又不属于最近原文窗口”的楼层。
- 动态召回：只检索 Godlog 细节，不再召回已经固定注入的锚点或合并块。
- 最近正文：由 SillyTavern 原生消息数组保留，默认最近 3 个 AI 回合。

## 3. 角色成长纪要为什么会混入玩家

旧实现有三个问题叠加：

1. “只追踪 角色甲”只存在于自然语言提示词，模型违例后没有代码校验。
2. `refreshCodexFromPatch()`会原样接受副 API 返回的人物纪要表；只要模型多写了“玩家名”，该行就被永久保存。
3. 设置页中的人物动态、人物库、物品规则虽然能保存，但后台 `buildCodexPatchPrompt()`和完整重建提示词没有真正读取这些设置，用户看到的规则并未实际生效。

0.9.1 改为双层约束：

```text
提示词白名单
   +
解析提交前代码白名单
```

`sanitizeCharacterMemoSection()`只按照 `trackedCharacterNames()`逐人重建表格：

- 输出行数 = 追踪白名单人数；
- 角色名强制使用白名单中的规范名称；
- 玩家名永远不能成为人物纪要行；
- NPC/配角永远不能成为人物纪要行；
- 模型漏写某个主角时，保留该主角上一版，而不是用玩家补位；
- 旧聊天首次加载时也会执行迁移，已经混入的玩家行立即清除。

出场人物库使用相反的边界：它只保留 NPC/配角，并过滤玩家及所有追踪主角。

## 4. 多人卡如何确定追踪对象

0.9.1 使用确定性规则，不让模型自己猜：

### 普通单角色卡

追踪 SillyTavern 当前 `name2`，也就是当前 `{{char}}`。

### SillyTavern 原生群聊

读取当前群组的活动成员，按角色记录/头像/成员 ID 解析为角色名，自动形成追踪白名单。用户在设置中额外填写的角色也会并入，不会因临时启用新成员而丢失。

### 一张角色卡内写了多个主角

SillyTavern 的 `name2`只能表示整张卡的显示名，无法可靠知道卡内究竟有几个独立主角。因此必须在“人物动态 → 人物纪要追踪角色名单”中逐行填写：

```text
角色甲
第二位主角名
第三位主角名
```

此时该名单是权威白名单；代码不会再根据正文里谁出现得多而擅自增删主角。

### 玩家排除

无论自动还是手动模式，当前 `name1` / `{{user}}`都会在名单标准化阶段和表格提交阶段被排除。

## 5. 动态召回重构

### 旧问题

- Prompt Ready 阶段直接等待向量 API，会让点击发送到请求真正发出之间出现停顿。
- “至少召回 N 条”会在没有相关旧事时强行塞入无关内容。
- 锚点、累计合并、逐楼摘要同时作为候选，会造成同一事件重复注入和旧状态复活。
- 每次关键词召回都会重新切分所有历史 Godlog。

### 新流程

1. `USER_MESSAGE_RENDERED`后立即构造“最新用户输入 + 最近 6 条可见上下文”的查询。
2. 向量模式在后台预取，主模型仍可继续走生成准备流程。
3. Prompt Ready 时绝不等待网络：
   - 预取已完成：使用向量结果；
   - 预取尚未完成或失败：立即使用关键词结果。
4. 只从有效 Godlog 中召回详细旧楼，不召回 merge/anchor 本体。
5. 排除最近原文窗口、已经在“尚未锚定逐楼摘要”中确定性注入的楼层。
6. 使用自适应阈值和总 Token 预算；相关度不足允许返回 0 条。
7. Godlog 关键词集合按“记录 ID + 正文哈希”缓存，旧摘要不变时不重复分词。

## 6. 对标 Horae 的性能策略

0.9.1 采用与 Horae 当前流畅热路径一致的原则，并针对 Anchor 的分层记忆增加了更严格的非阻塞边界：

| 热路径 | 旧 Anchor | 0.9.1 |
| --- | --- | --- |
| 流式 token | 每 token 清空缓存并全聊天重扫 | 240ms 节流，只探测末楼 |
| 消息完成 | 多事件各自排队 | 按消息键、正文哈希和任务状态合并 |
| Prompt 组装 | 可能等待向量网络 | 预取；Prompt Ready 永不等待网络 |
| metadata 读取 | 每次调用执行迁移、历史修复、覆盖重建 | 每聊天只标准化一次，热路径直接返回对象引用 |
| 关键词召回 | 每轮重新分词全部旧摘要 | 按摘要哈希缓存关键词集合 |
| Prompt 裁剪 | 每个旧楼重复扫描整个 Prompt | 每次生成建立一次文本索引 |
| 卡片 UI | 滚动反复删除重建 | 渲染签名 + 最近楼/视口附近虚拟化 |
| 后台工作台 | 关闭时仍构建全页面 | 关闭时不构建隐藏列表 |
| 保存 | 多次关键保存可能连续序列化整聊 | 约 900ms 合并，页面离开时 flush |
| 人物纪要 | 依赖模型遵守 | 提示词 + 代码白名单 + 事务替换 |

不能仅凭静态代码保证在每台设备上“绝对快于 Horae”，但 0.9.1 已移除 Anchor 已确认的逐 token 全历史重扫、Prompt 阶段网络等待、重复分词和反复数据迁移等主要热路径。

## 7. 编辑、Swipe、删楼和回退

每条 Godlog 绑定：

- 消息键；
- AI 楼层；
- 当前回合原文哈希；
- 更新时间和处理状态。

源正文发生变化时：

1. 旧 Godlog 立即退出有效注入链，但卡片可保留为“待刷新”。
2. 包含该楼的 15 回合锚点被移除。
3. 依赖该锚点的累计合并及后续累计合并被级联移除。
4. 人物/物品/场景索引保留最后有效快照，并标记待安全重建。
5. 人物关系先按每 10 个 AI 回合检查点 + 中间差量回放到目标楼，再用当前有效 Godlog 完整复核。
6. 新摘要与新索引生成并校验成功后，才事务式替换旧快照。

因此从第 68 楼删除回第 23 楼时，关系状态不会继续携带第 24—68 楼的变化。

## 8. 存储边界

- 聊天 metadata：Godlog、锚点、累计合并、紧凑状态索引、关系检查点/差量、向量引用和短注入记录。
- IndexedDB：Embedding 浮点向量主体。
- 运行时内存：聊天楼缓存、Godlog 索引、关键词集合、可视卡片签名、待完成召回预取。
- 不再为每个 AI 楼层复制一整份主模型注入正文，只记录来源引用、哈希、字符数和短预览。

## 9. 升级行为

从旧版本升级后：

- 保留已有 Godlog、15 回合锚点、100 回合累计合并、人物关系和状态索引；
- 旧人物纪要会按当前追踪白名单清洗，玩家/NPC 行移除；
- 旧出场人物库迁移为 5 列，过滤玩家和主角；
- 旧物品表迁移为 3 列；
- 向量 ID 和 IndexedDB 命名空间不变；
- 只有在用户改变追踪名单或来源剧情发生变化时，才进入安全重建。
