# Anchor Memory 0.9.4 摘要生命周期审计

## 用户报告现象

某楼已经存在逐楼摘要，但在该楼记录主模型注入、重新渲染或补充宿主元数据后，楼层灰色卡片显示“摘要缺失/待刷新”。打开摘要页仍能看到旧摘要，随后后台又可能自动重跑。

## 根因

存在两条互相叠加的判定：

1. 消息身份优先采用 `send_date/message_id`，宿主后补或规范化这些字段时，同一楼会从旧键切换成新键，导致楼层卡片找不到旧 Godlog；
2. 已完成摘要的 `rawHash` 与当前楼签名不同时，会被自动标记为 stale，并撤销其锚点、索引和注入资格。

因此数据正文没有真正消失，但 UI 和任务队列把它当成了缺失记录。

## 0.9.4 修复

- 消息始终采用 `anchor_memory_meta.stableMessageKey`，宿主 ID 只用于首次生成确定性种子；
- 数据版本与源签名架构升级，旧键按楼层兼容迁移；
- 已完成且有正文的 Godlog 被定义为持久快照；
- 后续楼层签名变化仅写入 `sourceMismatch/currentRawHash`，不更改 `status=ready`，不设置 stale；
- 不撤销锚点、不删除向量、不回滚人物关系、不标脏状态索引、不进入自动补写队列；
- 状态索引使用 Godlog 自己的 `rawHash` 作为版本签名；
- 只有用户手动重跑或手动保存修改，才更新摘要源签名并触发派生记忆回滚；
- 手动重跑先生成候选摘要，成功后才撤销旧派生记忆并提交；失败时旧摘要与旧锚点保持有效。

## 结果

旧摘要不会再出现“正文仍在但卡片显示缺失”的假缺口。真实未生成或首次生成失败的楼层仍保持原自动补写逻辑。
