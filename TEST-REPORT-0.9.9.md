# Anchor Memory 0.9.9 测试报告

## 测试目标

本轮重点验证两件事：

1. 在 SillyTavern 的最终提示词事件未触发、触发较晚或使用不同事件载荷形状时，Anchor Memory 是否仍能在发送前得到一份记忆，并且最终请求不会重复注入。
2. 插件暂停后是否真正停止工作、保留数据并清除提示词/隐藏楼层副作用；重新启动后能否继续原有记忆链路。

## 覆盖的异常条件

- 1–240 条对话消息与不同注入深度；
- 10 种最终提示词事件载荷形状；
- 最终提示词事件正常触发、完全不触发、插件已暂停；
- 100–360 回合长对话；
- 摘要延迟完成、最多 8 个永久缺失摘要；
- 运行中最多 4 次修改锚点/累计合并间隔；
- 语义召回在 1,800 ms 窗口内返回或超时；
- 暂停前存在待处理任务、提示词和插件管理的隐藏楼层。
- 在动态召回等待中暂停、以及宿主正在应用隐藏状态时连续切换开关。

## 注入链路结论

0.9.9 使用双层防护：

1. `generate_interceptor` 先写入 `setExtensionPrompt` 保底块，并执行旧正文窗口裁剪；
2. `CHAT_COMPLETION_PROMPT_READY` 若在本轮触发，会从真实最终消息数组删除所有 Anchor Memory 旧块，等待受限动态召回，再插入一份最新系统记忆。

因此，最终事件缺席时仍有保底；最终事件存在时会覆盖保底而非叠加。暂停状态则会主动清除两条路径中的 Anchor Memory。

## 范围说明

以上是对插件逻辑的本地确定性与随机压力模拟，并非在真实 SillyTavern 中发起 1,200 次联网模型请求。真实环境仍可能受到第三方扩展修改最终提示词、非标准后端实现或宿主版本差异影响；0.9.9 的双路径设计已将“仅因最终提示词事件未触发而整轮漏注入”的风险消除，并保留最终数组去重保护。
